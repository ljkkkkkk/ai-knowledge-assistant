# Lesson 02: 项目结构和 README

## 这一课要解决什么

上一课我们完成了两件事：

```text
本地有项目存档
GitHub 上有远程仓库
```

这说明项目已经“有家”了。

但一个项目光有家还不够。现在它更像一个刚租下来的空房间，里面只有几张桌子，还没有分清哪里写代码、哪里放资料、哪里放测试、哪里写说明。

Lesson 02 要做的事就是：

```text
给项目做一次房间布局。
```

布局做好之后，你以后新增功能时就不会乱放文件。

## 为什么项目结构重要

很多新手项目看起来像这样：

```text
main.py
test.py
new_test.py
final.py
final2.py
demo.py
data.txt
```

刚开始还能跑，但过几天你自己都会忘：

```text
哪个文件是入口？
哪个文件还能用？
哪个文件只是临时测试？
```

项目结构的作用，就是提前给每类文件安排位置。

它不是形式主义。

它解决的是一个很实际的问题：

```text
项目变大之后，我还能不能找到东西？
```

## 这个项目现在的结构

我们把项目整理成这样：

```text
ai-knowledge-assistant/
  app/
    api/
    core/
    services/
  data/
    samples/
  docs/
  scripts/
  tests/
  README.md
```

你可以把它想象成一个小餐馆。

### `README.md`: 门面招牌

`README.md` 是别人打开 GitHub 仓库时最先看到的东西。

它像一家店门口的招牌和菜单。

一个好的 README 至少回答四个问题：

```text
这个项目是什么？
它解决什么问题？
现在做到哪一步了？
怎么运行或继续开发？
```

如果 README 写得乱，面试官会觉得：

```text
这个项目可能只是随手堆出来的。
```

如果 README 写得清楚，即使项目还没完全做完，别人也能看出你在认真做工程。

### `app/`: 后厨

`app/` 是真正写后端代码的地方。

以后 FastAPI 的主程序、接口、业务逻辑都会放在这里。

它叫 `app`，不是因为它一定是手机 App，而是因为它代表 application，也就是“应用程序”。

### `app/api/`: 点菜窗口

`api/` 以后放接口。

接口可以理解成外界和程序说话的窗口。

比如之后我们会有：

```text
POST /documents/upload    上传文档
POST /questions/ask       向知识库提问
GET  /health              检查服务是否活着
```

用户不会直接摸到你的数据库，也不会直接调用文档解析代码。

用户只通过 API 和系统交流。

所以 `api/` 就像餐馆的点菜窗口：

```text
客人只负责点菜，后厨怎么做菜不用暴露给客人。
```

### `app/core/`: 配电箱

`core/` 以后放项目的基础配置。

比如：

```text
数据库地址
模型 API Key
日志格式
上传文件大小限制
```

这些东西不属于某个具体功能，但整个项目都会用到。

所以它像餐馆的配电箱和水管系统：

```text
平时不显眼，但一出问题全店都受影响。
```

### `app/services/`: 真正干活的人

`services/` 以后放业务逻辑。

什么叫业务逻辑？

就是这个项目真正要做的事：

```text
解析文档
切分文本
生成向量
检索片段
调用大模型
整理回答
```

`api/` 负责接收请求，`services/` 负责真正干活。

这样分开之后，代码会更清楚：

```text
api/       管外面怎么访问
services/ 管里面怎么处理
```

### `data/`: 仓库

`data/` 放本地数据。

比如示例文档、临时上传文件、后面生成的向量索引。

但注意：不是所有数据都应该上传到 GitHub。

比如用户上传的真实文件、模型密钥、很大的索引文件，一般不应该提交。

这也是 `.gitignore` 存在的原因。

### `data/samples/`: 样品区

`samples/` 放可以公开的小样例。

比如我们之后可以放一个短短的课程介绍文档，用来测试：

```text
系统能不能读文档？
能不能回答问题？
能不能给出引用？
```

样例数据很重要，因为它能让别人一拿到项目就知道怎么试。

### `docs/`: 项目日记

`docs/` 不只是放教程。

它以后还可以放：

```text
项目设计
踩坑记录
评测结果
面试复盘
功能路线图
```

对你来说，`docs/` 还有一个额外作用：

```text
防止学了忘。
```

每次学完，不是靠脑子硬记，而是把理解写进项目。

### `scripts/`: 工具箱

`scripts/` 放一些辅助脚本。

比如：

```text
批量解析测试文档
运行评测集
检查环境变量
清理本地临时文件
```

这些脚本不是主程序，但能帮我们减少重复劳动。

### `tests/`: 质检区

`tests/` 放测试代码。

测试的意义不是为了显得高级，而是为了帮你确认：

```text
我刚才改的东西有没有把原来能跑的东西弄坏？
```

以后项目变大，如果没有测试，你每改一次代码都像摸黑走路。

有测试，就像每次出门前检查钥匙、手机、钱包。

## 今天新增了什么

这一课会新增几个空目录：

```text
app/api/
app/core/
app/services/
scripts/
tests/
```

空目录本身不能被 Git 直接记录，所以我们会在里面放一个 `.gitkeep` 文件。

`.gitkeep` 没有什么魔法。

它只是一个占位符，意思是：

```text
这个目录现在还空着，但我希望 Git 先记住它。
```

## 今天你需要记住的核心思想

不要一上来就追求复杂架构。

我们现在做的结构只解决一个问题：

```text
让未来的代码有地方可去。
```

以后写上传接口，就放进 `app/api/`。

以后写文档解析，就放进 `app/services/`。

以后写配置读取，就放进 `app/core/`。

以后写测试，就放进 `tests/`。

这样项目会慢慢长大，而不是慢慢变乱。

## 今天的验收标准

完成之后，你应该能做到：

1. 看懂项目里每个一级目录大概负责什么。
2. 知道 `README.md` 是 GitHub 仓库的门面。
3. 知道 `api/`、`services/`、`core/` 的区别。
4. 知道为什么空目录里要放 `.gitkeep`。
5. 能用自己的话解释：项目结构不是装饰，而是为了以后不乱。
