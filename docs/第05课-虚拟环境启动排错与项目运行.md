# 第05课：虚拟环境启动排错与项目运行

## 这一课为什么出现

今天本来只是想自己启动 FastAPI 服务，但中间遇到了一个很典型的 Windows + Python 虚拟环境问题：

```powershell
d:\workshop\projects\ai-knowledge-assistant\.venv\Scripts\activate
```

PowerShell 提示：

```text
无法将 ...\Scripts\activate 识别为 cmdlet、函数、脚本文件或可运行程序
```

后来又尝试：

```powershell
.\.venv\Scripts\Activate.ps1
```

仍然找不到。继续检查后发现，`.venv\Scripts` 里面有 `python.exe`、`pip.exe`、`uvicorn.exe`，但是缺少激活脚本：

```text
Activate.ps1
activate.bat
activate
deactivate.bat
```

这说明虚拟环境主体存在，但激活脚本不完整。

## 一开始为什么会报错

在 PowerShell 里，不能直接执行：

```powershell
d:\...\.venv\Scripts\activate
```

这是因为 Windows 下不同终端的激活脚本不一样：

```text
PowerShell 使用 Activate.ps1
cmd 使用 activate.bat
Git Bash 使用 activate
```

所以 PowerShell 正确写法应该是：

```powershell
.\.venv\Scripts\Activate.ps1
```

这里前面的 `.` 和反斜杠表示：

```text
从当前目录开始找这个脚本，并执行它。
```

## 为什么又出现 Permission denied

后来尝试修复虚拟环境：

```powershell
python -m venv .venv --upgrade
```

结果出现：

```text
Permission denied: '.venv\Scripts\python.exe'
```

真正原因不是代码坏了，而是 `.venv\pyvenv.cfg` 里面记录的解释器路径有问题。当时它把虚拟环境自己的 Python 写成了升级来源：

```text
executable = D:\workshop\projects\ai-knowledge-assistant\.venv\Scripts\python.exe
command = ...\.venv\Scripts\python.exe -m venv --upgrade ...
```

大白话说，就是：

```text
它想让正在被使用的 python.exe 覆盖它自己。
```

Windows 不允许一个正在运行或被占用的 exe 文件被覆盖，所以报了 Permission denied。

## 最后怎么修好的

修复思路是：不要让 `.venv` 自己修自己，而是用外部 Python 修它。

实际使用的是外部 Anaconda Python：

```powershell
D:\software\tools\anaconda3\python.exe -m venv D:\workshop\projects\ai-knowledge-assistant\.venv --upgrade
```

这样 `.venv\pyvenv.cfg` 被改回了正确状态：

```text
executable = D:\software\tools\anaconda3\python.exe
command = D:\software\tools\anaconda3\python.exe -m venv --upgrade ...
```

但是 `--upgrade` 只更新了解释器，没有自动补齐激活脚本。因此又创建了一个临时 venv，用它生成标准的 Windows 激活脚本，再复制回当前项目的 `.venv\Scripts`：

```text
Activate.ps1
activate.bat
activate
deactivate.bat
```

最后删除了临时 venv。

## 修复后的验证结果

验证激活环境：

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\.venv\Scripts\Activate.ps1
python -c "import sys; print(sys.executable)"
```

输出确认使用的是项目自己的 Python：

```text
D:\workshop\projects\ai-knowledge-assistant\.venv\Scripts\python.exe
```

验证依赖：

```powershell
python -c "import fastapi, uvicorn; print('fastapi/uvicorn ok')"
```

验证项目入口：

```powershell
.\.venv\Scripts\python.exe -c "import app.main; print(app.main.app.title)"
```

输出：

```text
AI Knowledge Assistant
```

说明 FastAPI 应用可以正常导入。

## 以后启动项目的推荐方式

推荐方式一：先激活环境，再启动服务。

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\.venv\Scripts\Activate.ps1
uvicorn app.main:app --reload
```

推荐方式二：不激活，直接使用虚拟环境里的 Python 启动。

```powershell
.\.venv\Scripts\python.exe -m uvicorn app.main:app --reload
```

第二种方式更稳定，因为它明确指定：

```text
就用这个项目自己的 Python。
```

启动后访问：

```text
http://127.0.0.1:8000/docs
```

或者健康检查：

```text
http://127.0.0.1:8000/health
```

## 这一课要记住什么

1. `.venv` 是项目自己的 Python 小房间，依赖装在这里，不污染全局环境。
2. PowerShell 激活脚本是 `Activate.ps1`，不是 `activate`。
3. 找不到 `Activate.ps1` 时，先检查 `.venv\Scripts` 里到底有什么。
4. `Permission denied: python.exe` 常见原因是 Python 正在被占用，或者 venv 正在尝试覆盖自己。
5. 不激活环境也能运行项目，只要直接调用 `.venv\Scripts\python.exe`。

## 今天的小结

这次小插曲其实很有价值。它说明真实开发不是只写代码，还包括环境、路径、终端、依赖这些基础工程能力。

以后遇到类似问题，不要先怀疑业务代码，而是按顺序排查：

```text
当前在哪个目录？
使用的是哪个 Python？
.venv\Scripts 里有什么？
依赖能不能导入？
项目入口能不能导入？
服务能不能启动？
```

这个顺序比乱试命令靠谱得多。
